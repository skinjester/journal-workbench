2013-01-29 00:41:30
there are moments in life that you learn something new after hacking away at a problem for a while. as often as not, knowledge comes from removing a previously unquestioned assumption. it is a strange feeling when it happens - the feeling more profound than sex on the one hand, and the feeling of face.palm. The problem that I've been wrestling with for a couple of worksessions over the last couple of days was trivial in the end. there was a block of CSS embedded in the header-custom.php file (that is used on every page) using the typekit font state classes:
	.wf-loading {visibility: hidden};
	.wf-active {}
	.wf-inactive {}

the .wf-loading class was being applied to the <html> tag very briefly when typekit.load() was called, and then being unapplied when the fonts signaled they had loaded and the typekit active: event was fired. Causing. A. White. Flash. Problem Solved!

2013-01-29 10:50:01
noticed issues with unstyled font when reviewing last nights work on the machine in the office. Here is the sequence of events:
	- browser loads the new page
	- the new page renders using the default font and with the header and content blocks at their rest state
	- The document.ready() function is called
	- the typekit.load() is called and the default font visibly updates to the substitution font
	- the header transition function is called and the header/content blocks jump to the start state and animate to rest state

Here is the sequence of events as they should occur
	- browser loads the new page
	- the new page renders
		- the default font is hidden
		- the header/content blocks are hidden and placed in the animation start state
		- the document ready() function is called
		- the typekit load() function is called and the substitute font is enabled
		- the substitute font is made visible
		- the header transition function is called and the header/content blocks are made visible
		- the header/content blocks animate to the rest state
